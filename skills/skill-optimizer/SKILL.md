# Skill Design Reviewer

Skill 设计质量评审工具。**核心信念**：设计质量是语义问题，语义问题需要结构化引导而非自动化规则。

## 评价框架

每个 Skill 从 5 个维度评价，每个维度 🟢 通过 / 🟡 有瑕疵 / 🔴 不通过。

| # | 维度 | 核心问题 | 权重 |
|---|------|---------|------|
| 1 | **可触发性** | LLM 能否正确决定调用此 Skill？ | 高 |
| 2 | **可用性** | LLM 调用后能否顺利完成任务？ | 高 |
| 3 | **安全性** | 是否有不可逆伤害的风险？ | 高 |
| 4 | **内聚性** | 职责是否单一、边界是否清晰？ | 中 |
| 5 | **可维护性** | 后续迭代是否容易？ | 低 |

总评分规则：
- 🟢 **推荐**：全部维度 🟢，或仅有 🟡 `可维护性`
- 🟡 **有风险**：任一「高权重」维度为 🟡，或多个维度 🟡
- 🔴 **不推荐**：任一「高权重」维度为 🔴

## 评审流程

### Round 1: 表面检查

读取目标 Skill 的 `SKILL.md`，检查：

- **YAML frontmatter**：`---` 分隔的头部是否存在？是否包含 `name` 和 `description` 字段？
- **引用文件完整性**：SKILL.md 中引用的 `.py`/`.md`/`.yaml`/`.json`/`.sh`/`.ps1` 文件是否都存在？

若此轮发现 🔴（如缺少 frontmatter 或核心文件缺失），说明 Skill 有基本结构问题，先提示用户修复再进入深度评审。

### Round 2: 逐维度深入评审

读取 `references/probes/` 下对应的引导问题文件，按维度依次回答：

| 顺序 | 维度 | 引导文件 |
|------|------|---------|
| 1 | 可触发性 | `references/probes/1-triggerability.md` |
| 2 | 可用性 | `references/probes/2-usability.md` |
| 3 | 安全性 | `references/probes/3-safety.md` |
| 4 | 内聚性 | `references/probes/4-cohesion.md` |
| 5 | 维护性 | `references/probes/5-maintainability.md` |

每分析完一个维度，记录：
- **评级**：🟢 / 🟡 / 🔴
- **发现**：具体问题描述，注明 SKILL.md 中的位置
- **建议**：如何改进

后面维度的分析可以复用前面维度的发现。

### Round 3: 综合判断

汇总 5 个维度的评级，按权重计算总评分。
识别跨维度的交叉问题（例如：职责过宽导致 description 模糊，这是内聚性→可触发性联动）。
输出一句话总体判断。

### Round 4: 输出报告

直接生成以下 Markdown 报告并保存：

```markdown
# 📋 Skill 设计质量评审报告

**Skill**: {skill_name}
**日期**: {date}
**总评**: 🟢 推荐 / 🟡 有风险 / 🔴 不推荐

---

## 逐维度评级

| 维度 | 评级 | 概要 |
|------|------|------|
| 可触发性 | 🟢 | ... |
| 可用性 | 🟡 | ... |
| 安全性 | 🟢 | ... |
| 内聚性 | 🟡 | ... |
| 可维护性 | 🟢 | ... |

---

### 可触发性 🟢
- **{问题标题}** `SKILL.md:行号`

  - 💡 {改进建议}
```

## 评审原则

1. **以证据说话**：每个发现必须引用 SKILL.md 的具体段落或文件名，而非笼统判断
2. **承认上下文**：同一个模式在不同 Skill 中意义不同——"5 个步骤"在部署工具中是内聚的，在"工具合集"中可能是职责过宽
3. **可操作性**：每个 🔴/🟡 发现必须附带具体的改进建议，而非只说"不好"
4. **不当裁判当教练**：目标是帮助 Skill 作者发现问题，不是给 Skill 打分排名
