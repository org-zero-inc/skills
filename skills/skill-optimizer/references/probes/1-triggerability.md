# 维度 1：可触发性

核心问题：**LLM 能否正确决定调用此 Skill？**

## 1a. Description 的信号强度

读取目标 Skill 的 YAML frontmatter 中的 `description` 字段，分析：

- **开头**：是否以明确的语义开头（行为动词 / 领域名词），而非"做一些""这个是"等模糊开头？
- **触发词**：是否有 ≥3 个用引号包裹的触发关键词/短语？
- **信息密度**：是否在 2-3 句话内说清楚了"这个 Skill 做什么"？

摘录 description 原文，并评估信号强度。

## 1b. 边界声明

检查 SKILL.md 中是否有平台、包名、格式等约束条件（例如"仅 Android""仅 Windows""仅对某某包有效"）。

- 如果存在约束，description 中是否也声明了？
- 如果未声明约束，是否有被误触发的风险？

## 1c. 场景覆盖

- 是否有"也适用于""即使用户没有明确说"等模糊场景覆盖说明？
- 反向思考：**一个用户刚接触这个领域，会用什么词来求助？** 这些词在 description 中吗？
- 是否有明显应该触发但没写的场景？

## 评级标准

| 评级 | 条件 |
|------|------|
| 🟢 | description 信号强 + 边界声明清晰 + 场景覆盖充分 |
| 🟡 | 触发词不足 / 边界未声明 / 场景覆盖有一项缺失 |
| 🔴 | description 缺失或完全模糊 / 无任何触发信号 / 无边界声明但存在约束 |
