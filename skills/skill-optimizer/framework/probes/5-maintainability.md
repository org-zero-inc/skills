# 维度 5：可维护性

核心问题：**后续迭代是否容易？**

## 5a. 文件结构与大小

- 文件数量是否合理？（1 个 SKILL.md + 0~3 个脚本是典型范围）
- 单个文件是否过大？（SKILL.md > 200 行 或 脚本 > 500 行 时需留意）
- 是否遵循了渐进披露原则？（元信息 → SKILL.md → 脚本/引用，三级加载）

## 5b. 测试与验证

- 是否有 evals/evals.json 或测试用例？
- 如果有，测试覆盖了主要功能吗？
- 如果没有，是否可以通过手动验证？（低复杂度 Skill 不需要测试）

## 5c. 冗余与过时内容

- 是否有注释掉的代码、过时的说明、未完成的 TODO？
- 是否有"需要更新"类标记？
- 是否有重复的描述（同一信息在 SKILL.md 和 description 中重复写）？

## 评级标准

| 评级 | 条件 |
|------|------|
| 🟢 | 结构合理 + 有测试或无需测试 + 无冗余内容 |
| 🟡 | 文件偏大（接近阈值） / 无测试但值得有 / 少量冗余 |
| 🔴 | 文件过大 / 严重冗余 / 明显过时内容 |
