# Skill Journey Slides — Skill 编写旅程演示文稿生成器

本 Skill 用于总结一个 Skill 的编写过程，将散落在 session 上下文中的开发历程，提炼为
结构化的五阶段叙事，并调用 `frontend-slides` 生成精美的 HTML 演示文稿。

## 核心思路

Skill 的编写不是线性的代码堆砌，而是一段从困惑到清晰、从粗糙到精炼的旅程。本 Skill
从当前 session 上下文中提取关键事件，按五个阶段组织叙事，最终以视觉化的方式呈现。

## 工作流程

### 阶段 A：上下文收集与分析

仔细阅读当前 session 的完整对话历史，提取以下信息：

1. **触发背景** — 用户最初的请求是什么？遇到了什么问题或需求？
2. **关键决策** — 在编写过程中做了哪些关键选择？为什么？
3. **重要变更** — Skill 的方向、结构、内容有过什么调整？调整原因是什么？
4. **验证动作** — 有没有进行测试、评估、review？结果如何？
5. **最终成果** — Skill 的最终形态和核心能力是什么？

**分析方法：**
- 逐条扫描对话中的用户消息，识别需求表述、反馈意见、方向调整
- 关注工具调用记录（特别是文件写入/编辑），识别 Skill 的迭代版本
- 识别用户明确表达的"不对"、"换一个"、"加上"、"去掉"等调整信号
- 关注从模糊需求到清晰定义的演变过程

### 阶段 B：五阶段叙事构建

将提取的信息组织为以下五个阶段，每个阶段需包含：**标题、核心叙述、关键事件、引用原文**。

#### Stage 1：🔍 发现问题（Problem Discovery）

描述 Skill 要解决的问题是如何被发现的。

- **初始需求**：用户最初表达了什么？
- **痛点与困惑**：当前存在什么不便或缺陷？
- **机会识别**：为什么这是一个值得做 Skill 的事情？

**示例结构：**
> 在对话中，用户提出"我希望你能帮我写一个skill..."，这意味着用户需要一个自动化的方式来...
> 痛点在于：手动整理编写过程耗时且容易遗漏关键步骤...
> 机会：将编写过程结构化、可视化，让回顾和分享变得容易。

#### Stage 2：🔍 分析问题（Problem Analysis）

描述对问题的深入理解和拆解。

- **需求拆解**：将模糊需求拆解为具体的功能点
- **约束识别**：技术约束、用户偏好、环境限制
- **方案对比**：考虑过哪些不同方案？各自的优劣？
- **关键问题**：需要解决的核心技术或设计问题是什么？

**示例结构：**
> 通过与用户的对话，需求被拆解为：1) 上下文分析能力 2) 五阶段叙事框架 3) 演示文稿生成
> 约束：必须使用 frontend-slides 生成，必须跨平台...
> 关键问题：如何从散乱的 session 对话中提取结构化的叙事？

#### Stage 3：📄 Skill 计划（Skill Planning）

描述 Skill 的设计和规划。

- **架构设计**：Skill 的整体结构和流程
- **触发条件**：什么场景下应该触发此 Skill
- **输入输出**：预期的输入和输出格式
- **依赖关系**：需要哪些其他 Skill 或工具的协作

**示例结构：**
> Skill 设计为三阶段工作流：A) 上下文收集 B) 叙事构建 C) 幻灯片生成
> 触发关键词：总结skill编写过程、skill旅程、skill编写总结...
> 核心依赖：frontend-slides（用于生成 HTML 演示文稿）

#### Stage 4：🔧 Skill 调整（Skill Adjustment）

描述编写过程中的迭代和调整。

- **初始版本**：第一版是什么样的？
- **用户反馈**：用户提出了什么修改意见？
- **重要调整**：方向性变更、功能增减、结构重组
- **调整原因**：为什么要做这些调整？解决了什么问题？

**示例结构：**
> 初始版本采用了...但在与用户确认后发现...
> 关键调整：1) 输入来源从 git 改为 session 上下文分析 2) 风格从固定改为自适应
> 每次调整都使 Skill 更贴近用户的实际需求。

#### Stage 5：🧪 Skill 验证（Skill Validation）

描述验证和最终确认的过程。

- **验证方式**：如何确认 Skill 符合预期？
- **验证结果**：哪些方面通过了？哪些需要改进？
- **最终状态**：Skill 的最终形态和核心能力
- **经验总结**：从编写过程中学到了什么？

**示例结构：**
> 通过对 SKILL.md 的完整审查，确认了五阶段叙事框架的完整性...
> 最终 Skill 包含：上下文分析、叙事构建、幻灯片生成三大能力...
> 经验：从用户对话中提取结构化信息需要明确的阶段划分和引导。

### 阶段 C：演示文稿内容编排

将五阶段叙事转化为演示文稿的幻灯片结构：

| 幻灯片 | 内容 | 类型 |
|--------|------|------|
| 1 | 标题页：Skill 名称 + 副标题"编写旅程" | Title |
| 2 | 旅程概览：五阶段路线图 | Overview |
| 3-4 | 🔍 发现问题 | Content |
| 5-6 | 🔍 分析问题 | Content |
| 7-8 | 📄 Skill 计划 | Content |
| 9-10 | 🔧 Skill 调整 | Content |
| 11-12 | 🧪 Skill 验证 | Content |
| 13 | 关键收获 / 经验总结 | Closing |

**内容密度规则：**
- 每张幻灯片最多 1 个标题 + 4-6 个要点
- 要点使用精炼的短句，不要长段落
- 适当使用 emoji 标注状态（🟢🟡🔴🔧🔍📄🧪）
- 如果某个阶段内容特别丰富，可以增加幻灯片

### 阶段 D：调用 frontend-slides 生成演示文稿

1. **加载 frontend-slides Skill**，按其 Phase 1 的工作流进行内容确认
2. **传递幻灯片内容**给 frontend-slides：
   - Purpose: Internal presentation
   - Length: Medium 10-20
   - Content: All content ready（在阶段 C 中已编排好）
   - Editing: No（演示文稿无需内联编辑）
3. **风格选择**：让用户在 3 个预览中选择，推荐以下三个风格（根据内容特点）：
   - Creative Voltage（创意/技术混合）
   - Bold Signal（专业/自信）
   - Dark Botanical（优雅/深思熟虑）
4. 依次完成 frontend-slides 的 Phase 2（风格发现）→ Phase 3（生成）→ Phase 5（交付）

## 引用规范

在叙事中尽量引用用户在 session 中的原始表述：

- 直接引用用户原话，用 `>` 引用格式
- 标注引用来源（用户消息第几轮、什么上下文）
- 如果用户表述较长，可以提取关键短语

## 注意事项

- **客观呈现**：不要美化或夸张编写过程中的困难或成果
- **完整覆盖**：五个阶段都要有内容，不要跳过任何阶段
- **重点突出**：如果某个阶段内容特别少（如验证），诚实说明并简要带过
- **叙述连贯**：五阶段之间要有逻辑衔接，不是孤立的事件列表
- **中文输出**：所有叙述和幻灯片内容使用中文
