# Bug Fixer — 系统化缺陷修复工作流

本技能定义了一套严谨的缺陷修复流程，确保每次修复都治本而非治标、不引入新问题、并通过回归测试和同类模式排查防止再次发生。

## 核心原则

1. **证据驱动** — 不凭猜测修复，每一步都要有证据支撑
2. **治本不治标** — 修复根因，而非掩盖症状
3. **最小变更** — 只改必须改的，不夹带重构或新功能
4. **排查同类** — 修一处，查全局，相似问题一并处理
5. **防止回退** — 每个修复都配有回归测试

## 修复流程

### 第一步：复现

无法复现的 bug 无法确认修复。这是最不可跳过的步骤。

**执行要点：**
- 精确记录复现步骤（环境、数据、操作序列）
- 确认复现频率（100% / 间歇性 / 特定条件）
- 如果无法复现，先收集更多上下文（日志、监控、用户环境信息），而非猜测
- 在干净环境中验证，排除本地配置干扰

**必须回答的问题：**
- 这个 bug 在什么条件下必定发生？
- 有没有看似无关但实际上影响复现的因素？

### 第二步：定位根因

从现象逆推到根因，区分"症状"和"病因"。

**执行要点：**
1. **追踪代码路径**：从触发点沿调用栈/数据流逆推，找到精确的出错位置
2. **定位断点**：找到具体是哪一行/哪个函数导致了异常行为
3. **理解上下文**：为什么代码会走到这个错误分支？是数据问题、逻辑问题、还是边界未处理？
4. **检查近期变更**：用 `git log` / `git blame` 查看相关代码的最近修改，判断是否是回归
5. **二分缩小范围**：如果问题范围较大，用 `git bisect`、条件断点、注释排除法缩小范围

**必须回答的问题：**
- 根因是什么？是数据异常、逻辑错误、边界遗漏、还是外部依赖问题？
- 这个 bug 为什么之前没被发现？测试覆盖的盲区在哪里？

**警惕以下陷阱：**
- ❌ 把报错信息当作根因（报错是症状，不是病因）
- ❌ 在未确认的情况下假设原因
- ❌ 只看第一层调用栈就下结论

### 第三步：排查同类模式 ⚡

**这是本流程区别于普通修复的关键步骤。** 发现一个 bug，往往意味着相同的模式可能在其他地方也存在。

**执行要点：**
1. 提取当前 bug 的**失败模式**（如：空指针未校验、并发竞争、越界访问、类型不一致等）
2. 用搜索工具在代码库中全局搜索相同的模式
3. 评估每个命中点：是否存在同样的风险？
4. 将发现的问题一并修复，或在修复记录中标注供后续处理

**搜索策略示例：**
- 空指针 → 搜索同类变量的所有使用点，检查是否有相同的 null check 缺失
- 越界访问 → 搜索同类数组/集合操作，检查边界校验
- 并发问题 → 搜索同类共享状态的访问模式
- API 返回值不一致 → 搜索所有调用该 API 的地方

**必须回答的问题：**
- 代码库中还有没有同样的失败模式？
- 如果有，是现在一起修，还是记录下来后续处理？

### 第四步：实现修复

修根因，不修症状。最小变更，防御性编码。

**执行要点：**
- **修复根因**，而非在出错位置加 try-catch 掩盖
- **最小变更**：只改必要的代码，不在 bug fix PR 中夹带重构
- **防御性编码**：在修复点加入必要的校验/guard，但不过度防御
- **遵循现有模式**：看看项目中类似情况是如何处理的，保持一致性
- **类型安全**：如果语言支持，用类型约束防止同类问题

**禁止事项：**
- ❌ 在修复中顺便重构无关代码
- ❌ 添加新功能
- ❌ 过度防御（给不需要校验的地方加校验）

### 第五步：编写回归测试

**每个 bug fix 必须配有回归测试。** 这是防止同一个 bug 再次发生的最后防线。

**执行要点：**
1. **先写能复现原 bug 的测试**，确认修复前测试失败、修复后测试通过
2. 覆盖修复中发现的**边界条件**
3. 如果修复涉及多组件交互，编写**集成测试**
4. 更新已有测试中与新行为不符的部分

**测试覆盖要求：**
- ✅ 复现原始 bug 的测试
- ✅ 边界条件测试
- ✅ 正常路径仍然通过的测试
- ✅ 同类模式中其他位置的测试（如果第三步发现并修复了同类问题）

### 第六步：验证修复

证明 bug 已经真正修复，而非被掩盖。

**验证清单：**
- ✅ 按原始复现步骤操作，确认 bug 不再发生
- ✅ 测试修复周围的边界条件
- ✅ 确认没有引入新问题（运行全量测试）
- ✅ 在接近生产的环境中验证（如 staging）
- ✅ 验证同类模式的修复点也正常工作
- ✅ 检查性能是否有退化

### 第七步：沉淀

把修复过程转化为团队知识资产。

**执行要点：**
- **提交信息**：写清楚 bug 是什么、根因是什么、怎么修的、有没有同类问题
- **知识库**：如果是有代表性的问题，补充到团队知识库
- **防御检测**：考虑是否需要添加 lint 规则、类型约束等自动化手段防止同类问题
- **反思流程**：这个 bug 为什么没被更早发现？测试/代码审查/设计环节有什么可改进的？

**提交信息格式：**
```
fix(模块): 简要描述修复的问题

问题：[用户可见的症状描述]
根因：[根因分析，指向具体的代码位置]
修复：[修复方式的简要说明，为什么这样修]
同类：[是否排查了同类模式，发现了什么]

Closes #xxx
```

## 常见反模式

| 反模式 | 正确做法 |
|--------|---------|
| 猜测原因直接改代码 | 先复现，用证据定位根因 |
| 在报错处加 try-catch 吞掉异常 | 找到为什么异常会发生，修复根因 |
| 修复中顺便重构 | 只修 bug，重构另开 PR |
| 不写回归测试 | 每个 bug fix 必须配测试 |
| 只修这一处，不查类似问题 | 排查同类模式，一并处理 |
| "在我机器上是好的" | 在干净环境验证复现 |
| 修复后不跑全量测试 | 确认没有引入新问题 |

## 快速检查清单

修复完成后，逐项确认：

- [ ] bug 已稳定复现
- [ ] 根因已定位并记录
- [ ] 同类模式已排查
- [ ] 修复针对根因，非症状
- [ ] 变更范围最小化
- [ ] 回归测试已编写并通过
- [ ] 全量测试通过
- [ ] 提交信息包含完整上下文
